# Ceph: A Scalable, High-Performance Distributed File System

## 摘要

我们开发了Ceph，一个具有优秀的性能，可靠性，可扩展性的分布式文件系统。Ceph 通过将分配表替换为一种专为由不可靠对象存储设备（OSD）组成的异构和动态集群而设计的伪随机数据分布函数（CRUSH），最大化地实现了数据与元数据管理的分离。我们通过将数据复制，故障检测和恢复任务分发给运行着专用本地文件系统的 OSD，增加了 OSD 的智能化。一个动态的分布式元数据集群提供了及其有效率的元数据管理，并无缝适应广泛的通用和科学计算文件系统工作负载。性能测量显示在多种工作负载下，ceph有极其优秀的io性能和可扩展的元数据管理，可以支持元数据操作达到 250k/s。

### 1. 介绍

系统设计者一直以来都想提高文件系统的性能，因为它对于整个系统的性能非常关键。特别是科学界和高性能计算社区，推动了分布式存储系统在性能和可扩展性方面的进步，通常能提前几年预示更通用的需求。传统的解决方案，典型如 NFS，提供了一个非常直观的模型，在这个模型中 server 导出一个文件系统层次结构，客户端可以将其映射到本地的命名空间中。尽管应用非常广泛，client/server 模型固有的集中化已被证明是实现可扩展性的重大障碍。

更近期的分布式文件系统架构基于对象存储，原有的机械盘被智能的对象存储设备替代了，OSD 不仅仅是一块盘，它还包括了 CPU，网络和本地缓存包括底层的磁盘或者是RAID。OSD把传统的块层的接口替换成了可以读写更大的变长字节区间也就是对象，把底层块的分配交给了设备自己。client 通常和 metadata server 交互来执行 metadata 操作（open/rename），然后和 OSD 交互来读写，显著的提高了可扩展性。

系统在采取了这种模型后，还继续受到了可扩展性的影响，由于meta data 的分布式的限制。对于传统文件系统的持续依赖，比如分配列表和inode table，以及不愿意将只能下放给 OSD，进一步限制了可扩展性和性能，并增加了可靠性成本。

我们提出了 Ceph，一个分布式文件系统，它提供了卓越的性能和可靠性，同时承诺了无与伦比的可扩展性。我们的架构基于这样一个假设：PB级规模的系统本质上是动态的：大型系统不可避免地是增量构建的，节点故障是常态而非特例，并且工作负载的质量和特征会随着时间的推移而不断发生变化。

Ceph 通过消除文件分配表并将其替换为生成式函数，实现了数据和元数据的解耦。这使得 Ceph 能够利用 OSD 具备的智能特性，将围绕数据访问、更新序列化、数据复制与可靠性、故障检测及数据恢复等方面的复杂性下放到各个节点中。Ceph 采用了一种高度自适应的分布式元数据集群架构，极大地提升了元数据访问的可扩展性，并由此提高了整个系统的可扩展性。我们讨论了驱动我们在架构设计中做出诸多选择的目标与工作负载假设，分析了这些选择对系统可扩展性和性能的影响，并分享了我们在实现功能性系统原型过程中的经验。

### 2 系统概述

Ceph 文件系统有三个主要的组件：

* client，每个实例都提供了一个接近于 posix 的文件系统接口给 host 或者进程
* OSD 集群，存储了所有的数据和元数据
* 元数据集群，管理了文件系统的 namespace ，也协调安全性，一致性和缓存一致性\
  ![](<../.gitbook/assets/image (47).png>)

我们称 Ceph 的接口是类 posix 语义的，因为我们认为，为了更好地契合应用程序的需求并提升系统性能，适当地扩展接口并有选择放宽一致性语义是合理的。

架构的主要目标是：可扩展性（几百 PB 以上的数据），性能，可靠性。系统的可扩展性是从多个维度进行考量的，不仅包括系统整体的存储容量和吞吐量，还涵盖了针对单个客户端，单个目录或者单个文件的处理性能。我们的目标负载可能包括一些极端的case比如有数十个数百个hosts并发的读取或者写入同一个文件，或者同一个目录里创建文件。这种在运行于超算集群的科学应用中常见的场景，正日益预示着未来通用计算工作负载的发展趋势。更重要的是，我们认识到分布式文件系统的工作负载本质上是动态变化的，随着时间推移，活跃的应用程序和数据集的更迭，系统对数据和元数据的访问模式会出现显著的波动。Ceph 直接解决了可扩展性问题，并同时实现了高性能、高可靠性和高可用性，这主要归功于三个核心设计特性：数据与元数据的解耦、动态的分布式元数据管理，以及可靠性的自治式分布式对象存储。

**数据和元数据的解耦**：Ceph 最大化分离了文件的元数据管理和文件数据存储。元数据操作（open，rename等等）统一的被一个元数据集群管理，而客户端直接和OSD交互进行文件的读写。基于对象的存储长期以来一直期望把底层 block 的分配下放到底层设备来提高系统的可扩展性。然而，现有的基于对象的文件系统通常只是将冗长的，基于每个文件的块列表替换为较短的对象列表，而 Ceph 则彻底消除了分配列表。相反，文件数据被条带化（切分）到命名可预测的对象上，同时，一个名为CRUSH的专用数据分布函数负责将这些对象分配到存储设备上。这使得任何参与方都能通过计算得出构成文件内容的所有对象的名称和位置，从而彻底消除了维护和分发对象列表的必要性，简化了系统设计，并减轻了元数据集群的工作负载。

**动态的分布式元数据管理**：由于文件系统元数据操作在典型文件系统工作负载中占比高达一半，因此高效的元数据管理对系统整体性能至关重要。Ceph 利用了一种基于“动态子树分区”的新颖元数据集群架构，改架构能够自适应且智能地将管理文件系统目录层级的职责，分散到数十个甚至数百个MDS之间。层级式分区保留了每个MDS工作负载的局部性，这不仅促进了高效率的元数据更新，还为激进的预取提供了便利，从而大幅提升了常见工作负载下的系统性能。值得注意的是，元数据服务器之间的工作负载分配完全基于当前的访问模式；这使得Ceph能够在任何类型的工作负载下，有效地利用所有可用的MDS资源，并随着MDS数量的增加实现近乎线性的扩展。

**可靠的自治式分布式对象存储**：由成千上万台设备组成的大型系统本质上是动态的：他们以增量的方式构建，随着新存储设备的部署和旧设备的退役，系统规模会不断地扩张和收缩，设备故障是频繁且在预料之中的；同时，海量的数据在不断地被创建、迁移和删除。所有这些因素都要求数据的分布必须不断演进，以便有效地利用可用资源，并维持预期的数据副本冗余水平。Ceph 将数据迁移、数据复制、故障检测以及故障恢复的责任，全部下放到了实际存储数据的OSD集群；与此同时，在宏观层面上，这些OSD共同为客户端和元数据服务器提供了一个单一的逻辑对象存储池。这种方法使得Ceph能够更有效地利用每个OSD节点上自带的智能（CPU和内存），从而实现具备线性扩展能力的、可靠且高可用的对象存储。
